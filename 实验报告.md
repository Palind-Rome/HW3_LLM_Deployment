## 二、实验目的

本次实验在 ModelScope 魔搭平台上部署开源大语言模型，并针对中文语义歧义、指代消解、嵌套逻辑推理、多义词理解和课程助理应用场景进行问答测试。通过统一的问题集记录不同模型的回答，比较它们在理解准确性、推理能力、表达清晰度、稳定性和资源开销方面的差异。


然后编写推理脚本 `run_qwen_cpu.py`

```python
from transformers import TextStreamer, AutoTokenizer, AutoModelForCausalLM

model_name = "/mnt/data/Qwen-7B-Chat" # 本地路径
prompt = "请说出以下两句话区别在哪里？ 1、冬天：能穿多少穿多少 2、夏天：能穿多少穿多少"

tokenizer = AutoTokenizer.from_pretrained(
	model_name,
	trust_remote_code=True
)

model = AutoModelForCausalLM.from_pretrained(
	model_name,
	trust_remote_code=True,
	torch_dtype="auto" # 自动选择 float32/float16（根据模型配置）
).eval()

inputs = tokenizer(prompt, return_tensors="pt").input_ids

streamer = TextStreamer(tokenizer)
outputs = model.generate(inputs, streamer=streamer, max_new_tokens=300)
```

## 四、模型问答测试的结果记录与分析

本实验使用统一问题集，以保证不同模型之间可横向比较。

| 编号 | 能力维度 | 测试问题 |
| --- | --- | --- |
| Q1 | 语义歧义 | 请说出以下两句话区别在哪里？1、冬天：能穿多少穿多少 2、夏天：能穿多少穿多少 |
| Q2 | 语义歧义 | 请说出以下两句话区别在哪里？单身狗产生的原因有两个，一是谁都看不上，二是谁都看不上 |

| Q4 | 指代消解 | 明明明明明白白白喜欢他，可她就是不说。这句话里，明明和白白谁喜欢谁？请解释你的判断。 |
| Q5 | 多义词理解 | 领导：你这是什么意思？小明：没什么意思。意思意思。领导：你这就不够意思了。小明：小意思，小意思。领导：你这人真有意思。小明：其实也没有别的意思。领导：那我就不好意思了。小明：是我不好意思。请问：以上“意思”分别是什么意思？ |

### 4.1 Qwen-7B-Chat

结果截图：

![image-20260511144854634](screenshots/image-20260511144854634.png)

结果已保存到：`Qwen-7B-Chat/outputs/qwen_results.jsonl`
报告摘录版已保存到：`Qwen-7B-Chat/outputs/qwen_results.md`

结果分析：

Qwen-7B-Chat 在第一个“冬天/夏天能穿多少穿多少”的语义歧义问题上表现较好，能够结合季节语境解释同一句话在冬天表示“尽量多穿、注意保暖”，在夏天表示“尽量少穿、方便散热”。这说明模型能较好利用常识和上下文完成简单语义反转类理解。

但在第二个“谁都看不上”的歧义问题中，Qwen 没有识别出句子的核心区别。该问题的关键在于“谁都看不上”可以理解为“自己看不上别人”，也可以理解为“别人看不上自己”，也就是主客体方向不同。Qwen 将两句话判断为“意思相同，只是句子结构不同”，说明它对省略主语和宾语导致的中文歧义不够敏感。

在嵌套逻辑问题“他知道我知道你知道他不知道吗”中，Qwen 的回答出现了明显混乱。模型错误地把“他、我、你”对应为第一、第二、第三人称，并且没有稳定拆解出“你知道他不知道”“我知道你知道他不知道”“他是否知道这一层关系”等层级。它最终给出“不能确定”的结论，虽然保持了一定谨慎性，但推理过程并不可靠。

在“明明/白白”指代消解问题中，Qwen 判断为“明明喜欢白白”，但该句更合理的拆分是“明明明明白白白喜欢他”，即“明明清楚地知道白白喜欢他”，其中“他”指向明明，因此应理解为“白白喜欢明明”。Qwen 在这里没有正确完成断句和代词回指。

在“意思”多义词问题中，Qwen 能意识到“意思”存在多种含义，并给出了“想法/含义”“小意思”“有意思”“不好意思”等解释，整体方向部分正确。不过它没有逐句对应每一个“意思”的具体语境，也出现了“实际上”“对不起”等与原对话不匹配的解释，即产生了 AI 幻觉。因此 Qwen 对多义词有基本识别能力，但细粒度语境映射仍不够准确。

从运行耗时看，Qwen 的 5 个问题平均耗时约 347.57 秒。其中最快的是第二题，耗时 170.789 秒；最慢的是第五题，耗时 515.09 秒。CPU 环境下推理速度较慢，且耗时会受到回答长度、生成 token 数和问题复杂度影响。

总体来看，Qwen-7B-Chat 的优点是回答较简洁，在常识性语义问题上表现较稳；缺点是在复杂中文歧义、嵌套逻辑和指代消解中容易误判，尤其对中文中省略主语、断句和回指关系的处理不够细。

结果截图：

![image-20260511150719648](screenshots/image-20260511150719648.png)

运行输出文件：

结果分析：

ChatGLM3-6B 在第一个“冬天/夏天能穿多少穿多少”的问题上同样表现较好，能够准确指出两句话的差别来自季节语境，并分别解释为冬天“多穿保暖”、夏天“少穿散热”。这一题中，ChatGLM3 的解释清楚，和 Qwen 的结论基本一致。

在第二个“谁都看不上”的问题中，ChatGLM3 逻辑混乱，回答了重复的两句话，没有理解原文的语义信息。

在嵌套逻辑问题中，ChatGLM3 的回答较长，但存在明显编造式推理。它将句子改写成 A、B、C 三人的多层关系，并推导出“C 不知道 A 和 B 都知道的事情”等结论，但这些内容并不是原句直接表达的信息。原句的关键应是逐层分析“你知道他不知道”“我知道你知道他不知道”，以及最外层“他是否知道这一整件事”的疑问。ChatGLM3 没有沿着原句结构稳定推理，说明其复杂逻辑解析能力不足。

在“明明/白白”指代消解问题中，ChatGLM3 判断为“明明喜欢白白”，与 Qwen 一样没有正确断句。更合理的理解应是“明明明明白白白喜欢他”，也就是“明明清楚地知道白白喜欢他”，其中“他”指明明，所以应判断为“白白喜欢明明”。ChatGLM3 的回答还出现“明明知道白白喜欢她”的表述，与题干中的“他”不一致，说明其代词一致性也存在问题。

在“意思”多义词问题中，ChatGLM3 的回答较抽象，主要围绕“词语、句子、行为的含义”展开，没有结合对话中的具体语境逐一解释。例如“意思意思”应理解为“表示一下、送点礼或表达心意”，“不够意思”应理解为“不够朋友、不够讲情面”，“不好意思”则涉及“难为情、客气或歉意”。ChatGLM3 没有准确覆盖这些口语化含义，因此多义词语境理解表现弱于预期。

从运行耗时看，ChatGLM3 的 5 个问题平均耗时约 368.01 秒。其中最快的是第四题，耗时 145.795 秒；最慢的是第三题，耗时 606.397 秒。与 Qwen 相比，ChatGLM3 平均耗时略高，但单题耗时波动明显，尤其在嵌套逻辑问题上耗时最长。

总体来看，ChatGLM3-6B 的优点是回答篇幅较充分，遇到复杂问题时会尝试展开推理；缺点是推理过程容易偏离题干，在中文歧义、断句、指代和多义词解释上存在较多不准确之处。它的回答看起来更完整，但可靠性并不一定更高。

结果分析：

Baichuan2-7B-Chat 在第一个“冬天/夏天能穿多少穿多少”的问题上也能够正确识别季节语境差异。它指出冬天应尽量多穿以保暖，夏天应尽量少穿以保持凉爽，核心判断是正确的。不过回答中出现了“蔺温”这样的错字，说明生成文本的表面质量略有瑕疵，可能是 tokenizer 的问题。

在第二个“谁都看不上”的问题中，Baichuan 没有抓住主客体方向变化。它将题目理解为“第一句是概括性陈述，第二句是具体解释”，但题干真正考察的是同一句“谁都看不上”在不同隐含主语下可以表示“自己看不上别人”或“别人看不上自己”。因此，Baichuan 在复杂中文歧义识别上也没有答对。

在嵌套逻辑问题中，Baichuan 尝试把信息划分为“已知、未知和不确定”三个层次，这一点比完全散乱的回答更有结构感。但是它的层级拆分并不严格，回答中出现了“你知道他知道你知道他不知道”等题干之外的重组表达，最后也没有明确说明原句中“不知道”的对象和最外层疑问之间的关系。因此，它在复杂逻辑拆解上有一定分析意识，但推理链仍不稳定。

在“明明/白白”指代消解问题中，Baichuan 先说“没有明确说明谁喜欢谁”，随后又说“主要表达了明明喜欢白白”，前后不够一致。更合理的断句应是“明明明明白白白喜欢他”，即“明明清楚地知道白白喜欢他”，其中“他”指向明明，所以应判断为“白白喜欢明明”。Baichuan 没有正确完成断句，且回答末尾出现语句不通顺的问题。

在“意思”多义词问题中，Baichuan 的表现相对更好。它能够按序号列出多个“意思”的解释，并覆盖“询问意图”“不够友好或不够尊重”“有趣或令人好奇”“没有其他特殊意图”“不好意思表示道歉或尴尬”等语义。虽然它没有完整对应原对话中每一次“意思意思”“小意思”的具体口语含义，也有部分解释偏泛化，但相比 Qwen 和 ChatGLM3，它更接近逐项解释的要求。

从运行耗时看，Baichuan 的 5 个问题平均耗时约 1130.71 秒，明显慢于 Qwen 和 ChatGLM3。其中最快的是第四题，耗时 441.314 秒；最慢的是第五题，耗时 2781.8 秒。这是因为，该模型运行过程中出现过内存不足，我在实际测试时将加载参数调整为 `torch_dtype=torch.float16`、`low_cpu_mem_usage=True`、`offload_state_dict=True`。这些设置有助于降低模型加载阶段的内存峰值，使模型能够在受限 CPU 环境中跑起来，但也会带来更长的加载和推理时间；CPU 环境下 `float16` 也不一定能像 GPU 上一样带来速度优势。因此，Baichuan 的耗时偏高既与模型生成内容较长有关，也与内存受限下的加载策略有关。具体的内存限制解决方案我在”实验问题与解决方法“中进行了详细记录。

总体来看，Baichuan2-7B-Chat 的优点是回答结构较清晰，尤其在多义词问题中更愿意逐项解释；缺点是运行资源压力最大、耗时最长，在复杂歧义、断句和逻辑推理方面仍有明显错误。它适合作为横向对比中的第三个模型样本，但在本次 CPU 部署环境下使用成本最高。

| 部署稳定性     | 能直接运行，资源压力相对较小     | 能直接运行，耗时略高               | 初次运行内存不足，需改用低内存加载参数 | Qwen 和 ChatGLM3 更适合 CPU Notebook 体验，Baichuan 对资源更敏感 |

### 分任务对比

在简单语义歧义任务中，三个模型都能正确利用“冬天”和“夏天”的常识背景，理解同一句“能穿多少穿多少”在不同季节中的含义变化。这说明当前开源对话模型对常见生活常识和明确上下文有较好的处理能力。

在复杂中文歧义任务中，三个模型都没有准确解释“谁都看不上”的关键差别。该题真正考察的是中文里省略主语和宾语后，动作发出者与承受者可能发生变化。Qwen 认为两句含义相同，ChatGLM3 尝试区分但解释偏离，Baichuan 则把问题误解成概括句和解释句的关系。由此可见，模型面对高度依赖语境和隐含成分的中文句子时，仍然容易停留在表层字面。

在嵌套逻辑推理任务中，三个模型都没有给出严格可靠的逐层推理。Qwen 的人称理解和层级拆解混乱；ChatGLM3 虽然回答很长，但加入了题干没有的 A、B、C 关系，存在过度推理；Baichuan 尝试用“已知、未知、不确定”分类，结构更清楚一些，但仍未严格对应原句。因此，这类多层“知道/不知道”问题对三个模型都较难。

在指代消解与断句任务中，三个模型都未能正确判断“白白喜欢明明”。该题需要先断成“明明 明明白白 白白喜欢他”，再根据“她就是不说”推断“她”指白白，“他”指明明。三个模型都倾向于把句子直接理解成“明明喜欢白白”，说明它们对中文连续重复字造成的断句歧义处理能力不足。

在多义词理解任务中，Baichuan 的表现相对最好，因为它按照多个序号解释了“意思”在不同语境中的含义，虽然仍有泛化和漏项。Qwen 能列出部分常见含义，但没有逐句对应；ChatGLM3 的解释较抽象，更多停留在“词语或句子的含义”层面，未能充分解释“意思意思”“不够意思”“不好意思”等口语表达。

### 综合结论

综合三个模型的结果，Qwen-7B-Chat 在本次 CPU 部署中运行相对更快，回答简洁，适合处理简单语义理解问题；ChatGLM3-6B 的回答更长、更愿意展开解释，但在复杂问题中容易引入题干之外的信息；Baichuan2-7B-Chat 在多义词解释上相对更好，结构化程度较高，但部署时内存压力最大，实际运行耗时也最长。

从能力角度看，三个模型都能处理简单常识型问答，但在中文特有的歧义、省略、断句、指代和多义词精细解释上仍存在明显不足。若用于实际中文问答应用，简单问题可以直接参考模型输出；复杂语义和逻辑问题则需要人工复核。若从本次作业的部署体验角度比较，Qwen 和 ChatGLM3 更适合在 ModelScope CPU Notebook 中完成基础体验，Baichuan 能提供有价值的对比样本，但需要额外处理内存问题，部署成本更高。

## 七、结论

本次作业完成了在 ModelScope 魔搭 CPU Notebook 环境中部署和测试开源大语言模型的完整流程，包括平台登录与资源启动、Conda 环境配置、依赖安装、模型下载、推理脚本编写、问答测试结果保存以及项目仓库公开。通过这次实验，我对大语言模型从“模型文件”到“可运行问答程序”的部署过程有了更直观的认识，也体会到大模型实际运行时对计算资源、内存空间和依赖版本的要求较高。

在模型测试方面，我分别对 Qwen-7B-Chat、ChatGLM3-6B 和 Baichuan2-7B-Chat 进行了统一问题集测试。三个模型都能较好回答简单常识和语义场景题，例如都能解释“冬天/夏天：能穿多少穿多少”的语境差异；但在更复杂的中文歧义、嵌套逻辑推理、断句和指代消解问题上，三个模型都出现了不同程度的误判。这说明开源大语言模型虽然具备较强的语言生成能力，但面对中文中高度依赖语境、隐含主语和多层逻辑关系的问题时，仍然需要人工检查和修正。

从横向对比结果看，Qwen-7B-Chat 的回答相对简洁，CPU 环境下平均耗时较短，适合完成基础问答体验；ChatGLM3-6B 更倾向于展开解释，但在复杂问题中容易引入题干之外的信息；Baichuan2-7B-Chat 在多义词解释上结构更清晰，但部署时内存压力最大，运行耗时也明显更长。综合来看，不同模型在回答风格、推理稳定性和资源消耗上各有特点，不能只看模型是否能生成流畅文本，还需要结合具体任务、运行环境和输出可靠性进行评价。

此外，Baichuan2-7B-Chat 在初次运行时出现了内存不足并被系统杀死的问题，后续通过设置 `torch_dtype=torch.float16`、`low_cpu_mem_usage=True` 和 `offload_state_dict=True` 才顺利完成推理。这一过程说明，大语言模型部署不仅是调用代码接口，还需要根据硬件条件调整加载方式和运行参数。对于 CPU 免费资源环境而言，完成 7B 级别模型推理已经比较吃力，若要获得更好的速度和稳定性，通常需要更充足的内存或 GPU 资源。

总体而言，本次实验完成了 3 个推荐开源大语言模型的部署与问答测试，并通过截图、输出文件和横向分析记录了实验过程。实验结果表明，大语言模型在常识性中文问答中表现较好，但在复杂语义理解和严密逻辑推理方面仍存在明显边界；实际部署时也需要综合考虑模型大小、硬件资源、依赖环境和推理速度。通过本次作业，我不仅了解了大语言模型的基本部署流程，也对模型能力评估和部署成本之间的关系有了更具体的理解。
